# Глобальная карта: локации и переезд

Дата: 2026-06-17
Статус: утверждён, готов к плану реализации

## Контекст

PancakeFlip — прототип про жарку блинов (одна сцена `PancakeFlip.unity`, UI и сцена
генерируются `PancakeFlipSetup.BuildEverything()`). Прогрессия:
- `Wallet` (монеты/опыт/уровень) — создаётся заново в `Awake`, **без сохранения**.
- `OrderQueue` — пул заказов из набора рецептов; уже есть `AddRecipesToPool(...)`.
- `RecipeConfig` / `IngredientConfig` — ScriptableObject с полем `unlockLevel`.
- Счётчика «выполнено N заказов» сейчас нет; системы сохранения нет.

Цель: добавить мета-прогрессию «переезд между локациями». Выполнил N заказов в текущей
локации → открывается следующая; в новой локации другие покупатели, новый спрос и новые
рецепты/ингредиенты.

## Утверждённые правила (из брейншторма)

- **Модель контента — гибрид:** открытые рецепты и ингредиенты копятся в «книге» навсегда
  (доступны для готовки везде); покупатели и **спрос** (что заказывают) — локальные для
  каждой локации; новые рецепты/ингредиенты открываются при первом прибытии в локацию.
- **Карта — линейная цепочка + возврат:** локации идут по порядку; выполнил N заказов в
  текущей → открывается следующая; любую открытую можно посещать снова.
- **Условие открытия:** N сданных заказов в текущей локации (N задаётся по-локационно).
- **Переезд бесплатный** (после того как локация открыта).
- **Без сохранения:** прогресс карты session-only (как и вся игра сейчас).

## Подход

Локации — ScriptableObject-конфиги; переезд переключает контент **внутри той же сцены**
(пересборка `OrderQueue`, разблокировка рецептов/ингредиентов в книгу, смена покупателей).
Без загрузки отдельных сцен. Ложится на существующий паттерн «SO-конфиги + кодогенерация».

Рассмотрены и отклонены: отдельная сцена на локацию (тяжело, ломает одно-сценовость);
хардкод локаций в коде (негибко, против дата-ориентированного подхода).

## Принципы

- Чистая логика прогрессии карты (счётчики, гейтинг, валидность переезда) вынесена в
  тестируемый класс без зависимостей от сцены.
- Остаёмся в паттерне «всё генерит `BuildEverything()`»; новый UI — кодом, кнопка «Карта»
  адаптивна через существующий `ResponsiveRect`.
- Мобильный портрет и текущий геймплей не должны деградировать.

## Компоненты

### 1. `LocationConfig` (ScriptableObject) — одна локация

Поля:
- `displayName` (string);
- `mapIcon` (Sprite, опц.) — иконка узла на карте;
- `background` (Sprite, опц.) — зарезервировано под будущий арт локации (пока null);
- `ordersToUnlockNext` (int, N) — сколько заказов сдать здесь, чтобы открыть следующую;
- `demandRecipes` (RecipeConfig[]) — что заказывают здешние покупатели (пул `OrderQueue`);
- `unlockRecipes` (RecipeConfig[]) — добавляются в книгу при первом прибытии;
- `unlockIngredients` (IngredientConfig[]) — добавляются в магазин при первом прибытии;
- `customerSprites` (Sprite[]) — местные покупатели.

Ответственность: статически описывает одну локацию. Не содержит состояния.

### 2. `WorldMapConfig` (ScriptableObject) — карта

- `locations` (LocationConfig[]) — упорядоченная линейная цепочка.

Ответственность: задаёт порядок локаций. Единственный список, на который опирается
`MapState`.

### 3. `MapState` (рантайм, чистая логика — юнит-тесты)

Состояние:
- `CurrentIndex` (int) — где игрок сейчас;
- `HighestUnlockedIndex` (int) — докуда открыто;
- счётчики сданных заказов по индексу локации (массив/словарь);
- множество индексов, для которых уже применена разблокировка контента (чтобы не
  начислять повторно при возврате).

Конструктор принимает число локаций и массив `ordersToUnlockNext` (или ссылку на
`WorldMapConfig`-данные в виде простых значений, без UnityEngine-зависимостей).

Методы (чистые, детерминированные):
- `RecordOrderCompleted()` → инкремент счётчика текущей локации; возвращает `bool
  unlockedNew` (достигнут ли порог и открыта ли следующая впервые);
- `CanTravelTo(int index)` → `index` в диапазоне и `<= HighestUnlockedIndex`;
- `OrdersInCurrent()` / `OrdersToUnlockCurrent()` → для UI «сдано/N»;
- `ShouldApplyUnlock(int index)` → впервые ли мы открываем контент этой локации
  (для «копятся один раз»).

Событие `OnChanged` (для UI бейджа/прогресса). Без сохранения — живёт в памяти.

Ответственность: вся арифметика прогрессии карты. Не знает про ScriptableObject, сцену,
OrderQueue. Это и делает её тестируемой.

### 4. Интеграция в `GameSession`

- **Книга-аккумулятор:** множества открытых рецептов и ингредиентов, растут при прибытии в
  новую локацию (рецепты — в каталог/`OrderQueue`-источник через существующий
  `AddRecipesToPool`; ингредиенты — в доступный набор магазина).
- **`TravelTo(int index)`:**
  1. проверка `MapState.CanTravelTo`;
  2. если `MapState.ShouldApplyUnlock(index)` — добавить `unlockRecipes` в книгу и
     `unlockIngredients` в магазин (один раз);
  3. пересобрать `OrderQueue` из `demandRecipes` локации;
  4. сменить `customerSprites` у `CustomerAnimator`;
  5. (если `background` задан — сменить фон; пока no-op);
  6. сбросить активный заказ; обновить `MapState.CurrentIndex`.
- **В `TryServe`** после успешной сдачи: `MapState.RecordOrderCompleted()`; если открылась
  новая локация — событие для бейджа на кнопке «Карта».
- Старт: `MapState` инициализируется на локации 0; `TravelTo(0)` применяет её контент.

Замечание про `OrderQueue`: сейчас источник пула задаётся в конструкторе. Чтобы менять пул
на лету при переезде, добавим в `OrderQueue` метод `SetSource(RecipeConfig[])`
(сброс `_source`/`_pool`/`_visible` и `Refill`) — рядом с существующим
`AddRecipesToPool`. Это единственная правка существующего класса.

### 5. `MapScreenView` (UI, кодогенерация)

Полноэкранная модалка (как `RecipeBookScreen`/`UpgradeScreen`):
- цепочка локаций слева-направо (узлы); текущая подсвечена, закрытые — серые/замок;
- на текущем узле прогресс «сдано/N»;
- тап по открытому узлу → `GameSession.TravelTo(index)` + закрытие экрана;
- адаптивна: ограничивается центральной колонкой в ландшафте (как другие модалки).

Доступ: новая кнопка **«Карта»** в нижней панели (рядом с рецептами/апгрейдами), с
адаптивными якорями (`ResponsiveRect`, портрет/ландшафт). На кнопке — бейдж, когда
открылась новая локация.

### 6. Сборка (`BuildEverything`)

- Создать `LocationConfig`-ассеты для 3 сид-локаций (переиспользуя существующие 5 рецептов
  / 7 ингредиентов) и `WorldMapConfig` со списком.
- Создать `MapScreenView` (модалку) и кнопку «Карта» с обоими наборами якорей.
- Навесить/проинициализировать `MapState` и связать с `GameSession`.
- Никакого перехода на префабы; UI генерируется кодом.

### 7. Сид-данные (плейсхолдер)

3 локации, общий фон (арт позже):
- **Ларёк** — `demand`: обычный блин, блин с салями и сыром; `unlock`: тесто/салями/сыр;
  N=5.
- **Набережная** — `demand`: + блин с бананом и шоколадом; `unlock`: банан/шоколад; N=8.
- **Рынок** — `demand`: + грибной, клубника+шоколад; `unlock`: гриб/клубника; N=10.

(Точные наборы — деталь плана; используют уже существующие ассеты из `BuildEverything`.)

## Поток данных

```
Сдача заказа (GameSession.TryServe)
        v
MapState.RecordOrderCompleted() --(unlockedNew?)--> событие "новая локация" -> бейдж "Карта"
        |
Открыть "Карту" -> MapScreenView (список локаций, прогресс сдано/N)
        |
Тап по открытой локации -> GameSession.TravelTo(i)
        v
[книга += unlockRecipes/Ingredients (один раз)] -> OrderQueue.SetSource(demandRecipes)
        -> смена customerSprites -> сброс активного заказа -> MapState.CurrentIndex = i
```

## Краевые случаи

- Тап по закрытой локации — игнор (кнопка неактивна/серый узел), без исключений.
- Переезд в текущую локацию — no-op (или просто закрыть экран).
- Возврат в уже посещённую — контент не разблокируется повторно (`ShouldApplyUnlock`),
  спрос/покупатели пересобираются как у этой локации, счётчик прогресса сохраняется.
- N достигнуто, но игрок не уходит — продолжает выполнять заказы здесь; следующая остаётся
  открытой.
- Последняя локация в цепочке — `RecordOrderCompleted` не открывает ничего нового (нет
  следующей), без ошибок.
- Отсутствие `WorldMapConfig`/пустой список — система деградирует безопасно (одна локация,
  переезд недоступен).

## Проверка

Юнит-тесты (`MapState`):
- инкремент счётчика и открытие следующей по достижении N;
- `CanTravelTo` границы (нельзя в закрытую/за пределы);
- `ShouldApplyUnlock` срабатывает один раз на локацию;
- последняя локация не открывает несуществующую следующую.

Ручная проверка в Unity:
- старт на локации 0, спрос = её `demandRecipes`;
- сдать N заказов → на кнопке «Карта» бейдж, в экране карты открылась локация 1;
- переезд в 1 → новые рецепты в книге, новые ингредиенты в магазине, другой спрос/покупатели;
- возврат в 0 → контент не дублируется, спрос снова локальный для 0;
- портрет и ландшафт: кнопка «Карта» и модалка на местах.

## Вне scope этого цикла

- Арт локаций (фон/иконки), визуальное различие — поле `background` зарезервировано.
- Сохранение прогресса между запусками.
- Платный переезд, свободная (нелинейная) карта.
- Juice и тутор — отдельные циклы (тутор будет опираться на эту карту).
