# Адаптивный UI-фундамент: портрет (мобилка) + ландшафт (десктоп)

Дата: 2026-06-17
Статус: утверждён, готов к плану реализации

## Контекст

PancakeFlip — прототип игры про жарку блинов. Вся сцена и весь UI генерируются
одним editor-скриптом `PancakeFlipSetup.BuildEverything()`
(`Assets/Scripts/Prototypes/PancakeFlip/Editor/PancakeFlipSetup.cs`), плюс рантайм-вьюхи
с `Open()`/`EnsureUnderCanvas`. Всё захардкожено под **портрет 1080×1920** в двух
системах координат:

- **Мир** (ортокамера + `SpriteRenderer`): ширина считается как `camW = camH * 9/16`,
  стол масштабируется к `camW * 0.85`, фон/нижняя панель/покупатель привязаны к этому
  соотношению. `ortho = 5` (camH = 10), сцена привязана к нижней кромке.
- **UI** (`Canvas`, ScreenSpaceOverlay): `CanvasScaler` референс 1080×1920,
  `matchWidthOrHeight = 1` (по высоте). Якоря панелей подобраны под узкий-высокий экран
  (заказы слева 0–0.44 ширины, индикаторы прожарки справа, кнопки снизу).

В горизонтали (напр. 1920×1080) ломаются обе системы: стол раздувается по ширине,
UI-панели расползаются.

Цель: десктоп — **полноценная целевая платформа** (произвольный размер окна / fullscreen,
мышь). Механика подброса блина вертикальна по природе, поэтому широкий экран —
возможность: игровое поле остаётся вертикальной колонкой по центру, панели уезжают
в боковые поля.

Выбранный подход (из трёх рассмотренных): **«Разъехаться»** — один набор вьюх, у каждого
адаптивного элемента два варианта якорей (портрет/ландшафт), переключение в рантайме по
соотношению сторон. Меньше дублирования; будущие экраны (карта, тутор) наследуют тот же
механизм.

## Принципы

- Один источник правды об ориентации; компоненты подписываются, сами размер экрана не
  опрашивают.
- Остаёмся в текущем паттерне «всё генерит `BuildEverything()`». Не переходим на префабы.
  Скрипт лишь навешивает новые компоненты и проставляет оба набора якорей.
- Работает при живом ресайзе окна (десктоп), а не только на старте.
- Мобильный портрет визуально не должен деградировать относительно текущего состояния.

## Компоненты

### 1. `ResponsiveLayout` (ядро, рантайм-сервис)

Единый источник правды об ориентации.

- Определяет режим `Portrait` / `Landscape` по соотношению сторон экрана
  (`Screen.width / Screen.height`; порог — квадрат 1.0, с небольшим гистерезисом, чтобы
  не дёргалось на границе).
- Хранит «виртуальную игровую колонку»: фиксированный портретный слот 9:16 по центру
  экрана. Предоставляет её текущие метрики (ширина в world-юнитах и в долях экрана) для
  мира и UI.
- Отслеживает изменение размера экрана (poll `Screen.width/height` в `Update`, реакция
  только на фактическое изменение) и шлёт событие `OnOrientationChanged` (и при первом
  применении тоже).
- Singleton-доступ (как `GameSession.Instance`), `DefaultExecutionOrder` раньше
  потребителей.

Юнит-ответственность: «какой режим сейчас и какова игровая колонка». Не знает про
конкретные элементы UI или мира.

### 2. `ResponsiveRect` (UI-слой)

Компонент на `RectTransform` ключевых контейнеров. Хранит **два набора якорей**:

- `portrait`: `anchorMin`, `anchorMax`, (опц.) `pivot`, `anchoredPosition`/`offsetMin/Max`.
- `landscape`: то же.

Подписывается на `ResponsiveLayout.OnOrientationChanged` и применяет соответствующий
набор к своему `RectTransform`. На старте применяет текущий режим.

Элементы, которым назначаются оба набора (значения — на этапе плана/реализации):

- `OrderPanel` — портрет: поверх поля слева-сверху (как сейчас); ландшафт: колонкой в
  левом поле.
- `CookingIndicators` — портрет: справа поверх; ландшафт: в правом поле.
- Кнопки `RecipesBtn` / `UpgradesBtn` (контейнер `KitchenBottom`) — портрет: снизу;
  ландшафт: в правом поле под индикаторами.
- `TopBar` / `ProfileIcon` / `WalletIcon` — пере-якорятся под колонку.
- `ChargeIndicator` — под колонку.
- Модалки `RecipeBookScreen`, `IngredientsScreen`, `UpgradeScreen` — в ландшафте
  ограничиваются центральной колонкой (фикс. макс. ширина по центру), а не растягиваются
  на всю ширину; в портрете — как сейчас.

Юнит-ответственность: «применить нужный набор якорей к одному элементу». Не знает, как
определяется режим.

### 3. `WorldSceneFramer` (мировой слой)

Компонент, пересчитывающий раскладку мировой сцены от **реального** аспекта камеры.

- Игровая колонка (стол/сковорода/блин) держится фиксированного размера и по центру:
  стол масштабируется к ширине **виртуальной колонки** (из `ResponsiveLayout`), а не к
  реальному `camW`.
- Фон: оверскан покрывает полную ширину окна (логика `Max(camW/w, camH/h)` уже это умеет,
  пересчитать от реального аспекта).
- Нижняя панель-стойка: тянется на полную ширину окна (ОК визуально).
- Покупатель: точки входа/выхода offscreen — от реального `camW`.
- Вертикаль (`ortho`, привязка к низу) не трогаем.
- Пересчитывает на старте и при `OnOrientationChanged` / ресайзе.

Замечание: текущий `BuildEverything()` запекает масштабы/позиции в трансформы на этапе
билда под редакторный аспект. Фреймер переносит эти вычисления в рантайм (единый источник
правды), поэтому работает при любом размере окна.

Юнит-ответственность: «вписать мировую сцену в текущий кадр камеры, сохранив игровую
колонку». Не знает про UI.

### 4. `PancakeFlipScreenBootstrap` (обновление)

- Убрать принудительный портрет на standalone (сейчас при `Screen.width > Screen.height`
  ставит 1080×1920).
- Десктоп стартует в ландшафтном окне (напр. 1280×720), окно ресайзится; реальный ресайз
  ловит `ResponsiveLayout`.
- Мобилка — портрет, как раньше.

### 5. `BuildEverything()` (проводка, без смены паттерна)

- Создать объект-носитель `ResponsiveLayout` (или повесить на существующий
  bootstrap/session).
- Навесить `ResponsiveRect` на перечисленные UI-контейнеры и проставить оба набора якорей
  (портретные = текущие значения, ландшафтные = новые).
- Навесить `WorldSceneFramer` и связать с мировыми объектами/камерой.
- Никакого перехода на префабы — UI по-прежнему генерируется кодом.

### 6. Ввод

- Мышь уже работает: `PancakeFlipInputZone` использует `IPointerDownHandler` /
  `IPointerUpHandler`, `StandaloneInputModule` ловит мышь из коробки.
- Колесо для скролл-листов уже настроено (`scrollSensitivity`).
- Хоткеи / переключение fullscreen — **отложено** (договорённость «и то, и то позже»).

## Поток данных

```
Screen resize / start
        |
        v
ResponsiveLayout  --(режим + метрики колонки)-->  OnOrientationChanged
        |                                              |
        +--> WorldSceneFramer (мир: камера/спрайты)    |
        +--> ResponsiveRect × N (UI: якоря)  <---------+
```

## Обработка ошибок / краевые случаи

- Квадратное окно (аспект ≈ 1): порог с гистерезисом, чтобы не мигало; по умолчанию
  трактуем ≥1.0 как ландшафт.
- Очень узкое/широкое окно: игровая колонка ограничена по высоте (ortho), по ширине
  не превышает портретный слот; панели всегда в полях.
- Отсутствие `ResponsiveLayout` в сцене: компоненты `ResponsiveRect`/`WorldSceneFramer`
  деградируют безопасно (применяют портрет по умолчанию, без исключений).
- Ресайз во время открытой модалки: модалка пере-якорится на лету.

## Проверка (как убедимся, что работает)

- Запуск в портрете (1080×1920): визуально как сейчас, регрессий нет.
- Запуск в ландшафте (1280×720, 1920×1080): поле по центру вертикальной колонкой, заказы
  слева, индикаторы/кнопки справа, модалки по центру, мышь подбрасывает блин, скролл
  колесом работает.
- Живой ресайз окна туда-обратно: раскладка переключается без артефактов и без мигания
  на границе.

## Вне scope этого цикла

- Глобальная карта (мета-прогрессия, переезд между локациями).
- Обучение / тутор.
- Обратная связь (juice / реакции — требует отдельного уточнения смысла).
- Десктоп-специфичный арт, хоткеи, fullscreen-тогл.

Каждое — отдельным циклом дизайн → план → реализация поверх этого фундамента.
